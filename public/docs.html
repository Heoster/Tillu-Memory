Tillu-Memory API

Tillu-Memory is a cognitive memory layer for AI assistants. It separates working memory (what's happening right now) from long-term memory (what matters forever) and provides semantic search across all past interactions.

⚡
Working Memory
Upstash Redis · 30 min TTL
🧬
Semantic Memory
Supabase pgvector · cosine similarity
📌
Pinned Memory
Supabase Postgres · always injected

Authentication

All endpoints are server-to-server. There is no API key required on the HTTP level — access control is enforced by your backend. Each request must include a user_id which scopes all data to that user.

⚠️ Never expose this API directly to end users. Call it from your backend (Tillu-Think, etc.) only. All data is scoped by user_id — passing the wrong ID exposes another user's memory.

Base URL

https://tillu-memory.vercel.app

All endpoints accept and return JSON. Set Content-Type: application/json on all POST/PATCH/DELETE requests.

Error Handling

All errors return a JSON body with an error field and an optional detail or details field.

// 400 Bad Request
{
  "error": "Invalid request",
  "details": {
    "fieldErrors": { "user_id": ["Required"] }
  }
}

// 500 Internal Server Error
{
  "error": "Internal server error",
  "detail": "Supabase: relation does not exist"
}
400Missing or invalid fields — check the details.fieldErrors map
404Route not found
405Wrong HTTP method for this endpoint
409Consolidation already in progress for this user (lock held)
422Pin limit reached — unpin an existing memory first
500Upstream error (Supabase, Redis, HuggingFace, Groq)

Rate Limits

Limits are enforced per user_id.

Endpoint groupLimitWindow
Write operations (unified, write, forget, pin)30 requestsper minute
Read operations (read, search)120 requestsper minute
Consolidate1 requestper 10 minutes
Max pinned memories50per user (hard limit)
Max working messages50per session (oldest trimmed)

POST /memory/unified The magic endpoint — write + read in one call ▶

The primary interface for Tillu-Think. Send one request and get back everything needed to reason: the current conversation buffer, pinned facts, user profile, and semantically relevant past memories — all in a single response.

With action: "auto" (default), it writes the incoming message to working memory and reads all context layers simultaneously.

Request Body
FieldTypeRequiredDescription
user_idstringrequiredUnique identifier for the user
session_idstringrequiredCurrent conversation session ID
actionenumoptional"auto" | "read" | "write" — defaults to "auto"
messagestringoptionalThe incoming message text. Used as the write payload and semantic search query
roleenumoptional"user" | "assistant" | "system" — defaults to "user"
importanceenumoptional"critical" | "high" | "normal" | "low" — defaults to "normal"
Example Request
{
  "user_id":    "user_123",
  "session_id": "sess_456",
  "action":     "auto",
  "message":    "मुझे याद दिलाओ हमने API के बारे में क्या decide किया था",
  "role":       "user"
}
Response — 200 OK
{
  "working_memory": [
    { "role": "user", "content": "...", "timestamp": "2026-05-25T08:00:00Z" }
  ],
  "pinned_facts":  ["User's name is Rahul", "User prefers Hindi responses"],
  "relevant_past": [
    {
      "id":              "mem_abc",
      "content":         "Decided to use REST API with JSON format",
      "created_at":      "2026-05-24T10:00:00Z",
      "importance":      "high",
      "type":            "fact",
      "topic_tags":      ["api", "rest"],
      "similarity_score": 0.91
    }
  ],
  "profile": {
    "language_preference":  "hi",
    "communication_style":  "brief",
    "expertise_level":      "expert"
  },
  "actions_taken": ["write_to_working", "read_working", "read_pinned", "read_profile", "semantic_search"]
}
💡 action: "read" — skips the write, returns context only. Use this when Tillu-Think needs context before the user has sent a new message.
💡 action: "write" — only appends to working memory, no context returned. Use for assistant replies.
POST /memory/write Explicitly store a memory ▶

Store a specific memory with full control over type and importance. Use this when Tillu-Think detects a fact or preference worth persisting mid-conversation — for example, when the user says their name, mentions an allergy, or states a preference.

🔴 importance: "critical" → automatically sets is_pinned: true. No embedding is generated — the memory is always fetched directly on every request.
🔵 importance: "high" / "normal" / "low" → embedded via HuggingFace and stored for semantic search.
🟡 type: "preference" → also updates users.profile_data JSONB automatically.
Request Body
FieldTypeRequiredDescription
user_idstringrequiredUser identifier
contentstringrequiredThe memory text. Max 10,000 characters
typeenumrequired"fact" | "event" | "preference" | "summary"
importanceenumrequired"critical" | "high" | "normal" | "low"
session_idstringoptionalLinks the memory to a session for traceability
topic_tagsstring[]optionalTags for filtering, e.g. ["api", "deployment"]
Example — Critical Fact
{
  "user_id":    "user_123",
  "content":    "User's name is Rahul and he is allergic to peanuts",
  "type":       "fact",
  "importance": "critical",
  "session_id": "sess_456"
}
Response — 201 Created
{
  "memory_id":       "84b06e05-6d9a-449c-919f-7c4b20982a11",
  "is_pinned":       true,
  "profile_updated": false,
  "message":         "Memory stored and pinned (critical importance)."
}
GET /memory/read Read full context bundle — no writes ▶

Pure read — fetches working memory, pinned facts, user profile, and optionally runs a semantic search. No data is written. Used by Tillu-Voice to get language preference before speaking, or by dashboards to display user memory.

Query Parameters
ParamTypeRequiredDescription
user_idstringrequiredUser identifier
session_idstringrequiredSession to read working memory from
querystringoptionalIf provided, runs a semantic search and includes results
Example Request
GET /memory/read?user_id=user_123&session_id=sess_456&query=API+design
Response — 200 OK
{
  "working_memory":   [ ...last N messages... ],
  "pinned_facts":     ["User's name is Rahul"],
  "profile":          { "language_preference": "hi", "communication_style": "brief" },
  "semantic_results": [ ...top matching past memories... ]
}
POST /memory/search Semantic similarity search across all past memories ▶

Embeds the query using HuggingFace all-MiniLM-L6-v2 and runs a cosine similarity search against all stored memories for the user. Results are ranked by similarity with a time-decay boost — recent memories rank higher than old ones with the same score. Pinned memories are excluded (they're always returned via /memory/read).

Request Body
FieldTypeRequiredDescription
user_idstringrequiredUser identifier
querystringrequiredNatural language query. Max 1,000 characters. Supports Hindi, Tamil, English
top_knumberoptionalNumber of results to return. Default: 5, max: 20
time_filterenumoptional"all" | "last_7_days" | "last_30_days" | "last_90_days"
Example Request
{
  "user_id":     "user_123",
  "query":       "What did we decide about the database?",
  "top_k":       5,
  "time_filter": "last_30_days"
}
Response — 200 OK
{
  "results": [
    {
      "id":              "mem_xyz",
      "content":         "Decided to use Supabase Postgres for structured data",
      "created_at":      "2026-05-20T14:30:00Z",
      "importance":      "high",
      "type":            "fact",
      "topic_tags":      ["database", "supabase"],
      "similarity_score": 0.89
    }
  ],
  "query_used":  "What did we decide about the database?",
  "total_found": 1
}
POST /memory/consolidate Compress session into long-term memory ▶

The "sleep" function. Pulls all messages from Redis working memory, uses Groq (llama-3.3-70b-versatile) to extract a summary, key facts, and user preferences, then stores everything in Supabase and clears Redis. Call this when a session ends gracefully or from a cron job when the Redis TTL expires.

⏱️ This endpoint can take 5–15 seconds (Groq LLM call + multiple Supabase writes). It has a 30-second timeout on Vercel and a 10-minute rate limit per user. A Redis lock prevents duplicate processing — concurrent calls return 409.
Request Body
FieldTypeRequiredDescription
user_idstringrequiredUser identifier
session_idstringrequiredSession to consolidate
Example Request
{
  "user_id":    "user_123",
  "session_id": "sess_456"
}
Response — 200 OK
{
  "summary":               "User asked about Python deployment and decided to use Railway over Docker.",
  "facts_extracted":       3,
  "preferences_extracted": 1,
  "session_id":            "sess_456",
  "message":               "Session consolidated into long-term memory."
}
DELETE /memory/forget GDPR-compliant memory deletion ▶

Delete one specific memory by ID, or wipe an entire category of memories for a user. Designed for GDPR compliance and user-requested data deletion. Provide either memory_id or type — not both.

Request Body
FieldTypeRequiredDescription
user_idstringrequiredUser identifier
memory_idstringoptional*Delete one specific memory by ID
typeenumoptional*"all" | "working" | "semantic" | "pinned" | "profile"
session_idstringoptionalRequired when type = "working" or type = "all"

* Provide exactly one of memory_id or type.

Type values explained
typeWhat gets deleted
allEverything — Redis working memory + all Supabase memories + profile reset
workingRedis working memory for the given session only
semanticAll non-pinned memories from Supabase
pinnedAll pinned memories from Supabase
profileResets profile_data to {} and deletes preference-type memories
Example — Delete one memory
{
  "user_id":   "user_123",
  "memory_id": "84b06e05-6d9a-449c-919f-7c4b20982a11"
}
Example — Wipe everything (GDPR)
{
  "user_id":    "user_123",
  "type":       "all",
  "session_id": "sess_456"
}
Response — 200 OK
{
  "deleted": ["all_memories", "working_memory", "profile"],
  "message": "Deleted: all_memories, working_memory, profile."
}
PATCH /memory/pin Pin or unpin a memory ▶

Toggle the pinned state of a memory. Pinned memories are always injected into context on every /memory/unified and /memory/read call — they never need to be searched for. Maximum 50 pinned memories per user. Attempting to pin beyond the limit returns 422.

Request Body
FieldTypeRequiredDescription
user_idstringrequiredUser identifier
memory_idstringrequiredID of the memory to pin/unpin
pinnedbooleanrequiredtrue to pin, false to unpin
Example Request
{
  "user_id":   "user_123",
  "memory_id": "84b06e05-6d9a-449c-919f-7c4b20982a11",
  "pinned":    true
}
Response — 200 OK
{
  "memory_id": "84b06e05-6d9a-449c-919f-7c4b20982a11",
  "is_pinned": true,
  "message":   "Memory pinned. It will always be included in context."
}

Memory Layers

Five distinct layers, each with a different purpose and lifetime.

LayerStorageLifetimePurpose
⚡ WorkingUpstash Redis30 min inactivity TTLCurrent conversation buffer. Last 50 messages per session
📌 PinnedSupabase PostgresPermanentCritical facts always injected into context. Max 50/user
🧬 SemanticSupabase pgvector90 days (if unaccessed)Embedded memories. Similarity search. Conceptual recall
📊 ProfileSupabase JSONBPermanentAggregated user preferences — language, style, expertise
🗂️ SessionsSupabase PostgresPermanentCompressed session summaries. "What did we talk about last week?"

Integration Flows

Flow A — New message arrives (Tillu-Think)
1. User sends message
2. POST /memory/unified  { action: "auto", message: "...", role: "user" }
3. Response contains: working_memory + pinned_facts + profile + relevant_past
4. Tillu-Think builds prompt using this context
5. Tillu-Think generates reply
6. POST /memory/unified  { action: "write", message: reply, role: "assistant" }
Flow B — Session ends / user goes idle
1. No activity for 30 min → Redis TTL expires
2. Cron job (or Tillu-MAX) calls POST /memory/consolidate
3. Groq summarizes conversation → extracts facts + preferences
4. Summary stored in sessions table
5. Facts embedded → stored in memories table
6. Preferences merged into users.profile_data
7. Redis working memory cleared
Flow C — Critical fact detected
1. User: "मेरा नाम राहुल है और मुझे peanuts से allergy है"
2. Tillu-Think detects critical facts
3. POST /memory/write  { content: "...", type: "fact", importance: "critical" }
4. Memory auto-pinned → injected into every future context bundle
